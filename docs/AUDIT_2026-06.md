## 🔴 Critical Findings — Fix Now (10 items)

### Security

#### C-1: localStorage Role Bypass
**File:** `src/contexts/AuthContext.jsx:137,221` + `src/utils/permissionGuard.js:18-48`

Any user with DevTools can do:
```js
localStorage.setItem('platform_mock_user', JSON.stringify({...x, role:'admin'}))
```
…and every service-layer `requirePerm()` becomes a no-op. RLS is the only real defense.

**Fix:** Read role from JWT (`supabase.auth.getUser().user_metadata.role`) instead of localStorage. Or accept that RLS is the sole authority and remove all `requirePerm` calls (the current state is confidence theater).

#### C-2: permissionGuard fail-open by design
**File:** `src/utils/permissionGuard.js:42-47`

If profile is missing, role is missing, or role isn't in `ROLE_PERMISSIONS`, the guard returns `true`. Combined with C-1, deleting the localStorage key (or setting a custom role) defeats every service-layer guard.

**Fix:** Default to `false` for unknown roles/missing profile.

### Data Correctness (not just slowness)

#### C-3: CRM Dashboard shows wrong numbers for tenants > 1000 opportunities
**File:** `src/services/dashboardService.js:159-173`

`fetchOpportunityStats` uses `.select(...)` without `.range()` or `.limit()`. Supabase silent cap at 1000 rows. **Five derived useMemos** (`revenue`, `funnel`, `topPerformers`, `openValue`, `pipelineData`) all read from `rawOpps`. A manager with 5,000 opps sees ~20% of reality across the whole dashboard.

**Fix:** Convert to a Postgres RPC that returns aggregates (counts + sums per stage / per user).

#### C-4: Source Breakdown + Leads-by-User un-paginated
**File:** `src/pages/crm/CrmDashboardPage.jsx:1230,1283`

Same silent-cap bug. Top Performers numbers and Source Performance percentages are systematically wrong for any active tenant.

**Fix:** Postgres aggregate RPC (`GROUP BY lower(source)` etc).

#### C-5: ContactsPage stage filter fetches every opportunity client-side
**File:** `src/pages/ContactsPage.jsx:991-1044`

Full opportunities table scan on every stage filter / global filter change. Returns up to 1000 rows just to compute per-stage counts.

**Fix:** RPC `get_contact_ids_by_stage(p_stage, p_dept, p_agent)` returning just matching contact IDs.

### Data Integrity

#### C-6: Lead distribution not transactional
**File:** `src/services/contactsService.js:885-966`

`distributeLeadToAgents` loops N agents, inserts each clone separately, returns `{created, errors}`. No rollback if 4 of 5 clones succeed and the 5th hits RLS — orphans become real DB rows.

**Fix:** Wrap in an RPC (single transaction) like `permanent_delete_contact_rpc`.

#### C-7: createActivity → last_activity_at → task auto-complete not atomic
**File:** `src/services/contactsService.js:1152-1183` + `activitiesService.js:234-265`

Three sequential writes. If contact update succeeds but task-complete update fails, the timeline shows the call but the follow-up task stays "pending" forever.

**Fix:** Move to a DB trigger or RPC; today the failure path uses bare `catch {}`.

### Code Quality

#### C-8: Operations + Attendance still on mock data
**File:** `src/services/operationsService.js:6-10`, `src/services/attendanceService.js:3`

Live services mutate `MOCK_OPS_DEALS`, `MOCK_INSTALLMENTS`, `MOCK_HANDOVERS`, `MOCK_TICKETS` arrays in memory; Attendance reads from `attendanceStore` (localStorage). These modules **never migrated** off prototype state. Edits vanish on cache clear and don't sync between users.

**Fix:** Prioritize Supabase migration for these four modules, or hide their pages behind a feature flag.

#### C-9: Zero tests in the codebase
**File:** none

No `*.test.*`, `*.spec.*`, `__tests__/` anywhere. No Vitest/Jest in devDependencies. `payrollCalculator.js` (calculates real money), `phone validation`, `roleFilter.js`, `permissionGuard.js` — all completely untested. Any refactor is a lottery.

**Fix:** Add Vitest + first tests for the pure-function critical modules.

### Accessibility

#### C-10: 414 of 424 buttons missing aria-label + zero htmlFor in entire codebase
**Files:** repo-wide

Only 10 of 424 `<button>` tags carry an `aria-label`. Most icon-only buttons (Eye, Trash, Edit, ChevronUp/Down icons) have no accessible name. Zero `<label htmlFor=...>` exist — every `<label>` is visual-only, so clicking a label doesn't focus its input and assistive tech can't associate them.

**Fix:** Codemod pass — wrap each `<button>[icon-only]</button>` with `aria-label`; add `id`/`htmlFor` pairs to every form field.

## A. Security Audit (Full Report)

Scope: 88-table Supabase backend, React+Vite SPA on Vercel. RLS is the primary authority; this audit looks for client-side weaknesses, bypass paths, and audit gaps that erode defense-in-depth.

### A.1 Authentication & Session

#### 🔴 Critical — Profile mirrored to localStorage as the role source of truth
`src/contexts/AuthContext.jsx:137,221,246,364` writes the full profile (including `role`) to `localStorage.platform_mock_user`. `src/utils/permissionGuard.js:18-25,41-48` reads that same key and trusts the `role` field for every `requirePerm()` check. A malicious user with DevTools can simply do `localStorage.setItem('platform_mock_user', JSON.stringify({...x, role:'admin'}))` and every service-layer guard becomes a no-op (`hasPerm` short-circuits to `true` on line 44). RLS still protects the database, but every UI surface, route gate (`PermissionGate`), and impersonation feature trusts this value. **Fix:** re-derive role on every check via `supabase.auth.getUser().user_metadata.role` (server-signed JWT), or at minimum HMAC-sign the cached profile.

#### 🔴 Critical — `permissionGuard` is fail-open by design
`src/utils/permissionGuard.js:42-47`: if profile is missing, role is missing, or role isn't in `ROLE_PERMISSIONS`, the guard returns `true`. Combined with the localStorage bypass above, deleting the key (or setting `role: 'custom_role_xyz'`) defeats every service-layer guard. The doc comment explicitly says "RLS is the real authority" — but that puts 100 % of the security burden on RLS policies whose correctness this audit can't verify from JS alone.

#### 🟠 High — Auth state mutation race / forged login persistence
`src/contexts/AuthContext.jsx:108-121`: on every mount, `restoreMockSession()` is called even in Supabase-auth mode (line 197-201 is the mock branch, but the localStorage read happens at top before the env check). More importantly, `loginWithMock` (line 205-224) is still callable in production — `MOCK_USERS` is `{}` so it always throws, but if any future code path writes `platform_mock_user` directly, the `loading=false` snapshot UI grants access for ~1 frame before Supabase session validation completes. Tighten: in `USE_SUPABASE_AUTH` mode, never read `platform_mock_user` for session restore.

#### 🟠 High — Profile timeout silently downgrades to user_metadata
`src/contexts/AuthContext.jsx:23-78`: a 5-second timeout on the `users` table fetch falls back to `auth.user_metadata` (line 51-61), which a user *can* modify via `supabase.auth.updateUser({ data: {...} })` if the auth API allows it (default Supabase does — user_metadata is user-writable). If a deactivated user's `status='inactive'` lives in the `users` table but RLS denies them the read, the timeout fires, falls back to metadata (which still says `role:'sales_agent'`), and they're logged in. Fix: make the table read mandatory; on timeout/error, force sign-out.

#### 🟡 Medium — Logout doesn't clear all sensitive caches
`src/contexts/AuthContext.jsx:302-339`: only `platform_mock_user` + `platform_original_user` are removed. `platform_rbac_roles`, `platform_rbac_user_roles`, `platform_security_config`, `platform_notifications`, `platform_export_logs`, `platform_meta_ads_config` (contains Facebook API tokens), `platform_custom_fields` all persist to the next user on the same browser. Fix: enumerate `localStorage` and remove every `platform_*` key on logout.

### A.2 Permission Gates & RBAC

#### 🟠 High — Two parallel RBAC systems disagree
`src/services/rbacService.js` defines a second role system (`admin/manager/sales/hr/finance/viewer`) stored in localStorage, completely separate from `src/config/roles.js` (the real one). `getUserRole()` line 295-298 returns `'admin'` as default for any unknown userId. The Settings UI may surface this and let an operator believe they've restricted access when in reality the live system uses `roles.js`. Either delete `rbacService.js` or wire it to actually drive `permissions`.

#### 🟠 High — Permission overrides cached unrefreshed
`src/contexts/AuthContext.jsx:80-91`: `_permissionOverrides` is module-scoped and only reloaded on explicit `reloadPermissionOverrides()` call. If admin restricts a permission for `sales_director` mid-session, existing sessions keep the old (broader) permission set until they refresh. Listen to system_config realtime channel and reload on change.

#### 🟡 Medium — Mock impersonation persists role to localStorage in production builds
`src/contexts/AuthContext.jsx:342-377`: `impersonate()` writes the impersonated role to `platform_mock_user`. The function is reachable from the context (`isImpersonating, impersonate` are exported on line 399). With `MOCK_USERS={}` in production, `entry` would be `undefined` and it returns early — but the function still exists. If any Settings page calls `impersonate('admin')`, the localStorage profile is rewritten with admin role and every service-layer check now passes. Remove or gate behind `import.meta.env.DEV`.

### A.3 Service-Layer Guards — Unprotected Mutations

#### 🟠 High — `financeService.createJournalEntry` has no permission gate
`src/services/financeService.js:53-79` writes to `journal_entries` and `journal_entry_lines` (general ledger!) without `requirePerm(P.FINANCE_VIEW)` or similar. Anyone with a session can call this from DevTools. Same gap in `operationsService.createDeal/updateDeal/deleteDeal` (`src/services/operationsService.js:55-`) — no `requirePerm(P.OPS_MANAGE)`.

#### 🟠 High — `notificationService` has zero auth checks
`src/services/notificationService.js:162,187,204,220,235`: any logged-in user can insert arbitrary notifications, mark anyone's notifications read, or `clearAll()` which calls `delete().neq('id','')` — deletes *every* notification row visible to them (if RLS allows). Add `requirePerm` + scope deletes to `for_user_id=current_user`.

#### 🟠 High — `leadRecyclingService.runTemperatureDecay` unrestricted
`src/services/leadRecyclingService.js:23-65`: mass-updates contacts' temperature with no permission gate and no audit log. Any user could call it repeatedly to flip every hot lead to warm/cold.

#### 🟡 Medium — `expenseClaimService` approve/reject/markPaid take approverName as a string
`src/services/expenseClaimService.js:227,262,297`: no `requirePerm(P.FINANCE_VIEW)` or similar; `approverName` is whatever the caller passes. A sales_agent could claim "approved by CFO" with any string. Tighten: derive approver from session, not args; gate behind a permission.

#### 🟡 Medium — `viewTrackingService.logView`, `dashboardService`, `okrService`, `widgetService` lack guards
None of these enforce per-user scoping in JS (relies entirely on RLS). For OKRs, a malicious user can update/delete any objective they can see if RLS isn't perfectly scoped.

### A.4 Cross-User Data Leaks

#### 🟠 High — `notificationService.getNotifications` builds an unsafe OR
`src/services/notificationService.js:74-89`: `userId` and `userName` are accepted as args from the caller. If a caller passes nothing, the query has no `for_user_id` filter and returns notifications for everyone visible to RLS. Plus `for_user_id.eq.${userId}` is string-interpolated into a PostgREST `or()` — a userId of `all,(...)` could subvert intended scoping (PostgREST does parse parens). Fix: read userId from session only, never from arg.

#### 🟠 High — `applyRoleFilter` returns the unfiltered query on most "Other roles"
`src/utils/roleFilter.js:137`: `marketing/hr/finance` roles fall through to `return query` (unfiltered). Combined with marketing's `CONTACTS_VIEW_OWN` permission, the page expects scoping but the query returns everything RLS allows. This makes RLS the only line of defense; if any contacts RLS policy is permissive for hr/finance/marketing roles, they see all leads.

#### 🟡 Medium — Search and agent-name filters interpolate without quoting
`src/services/activitiesService.js:69`, `src/services/tasksService.js:75`, `src/services/auditService.js:42`, `src/services/expenseClaimService.js:127`, `src/services/contactsService.js:201`, `src/services/leadRecyclingService.js:79`, `src/services/notificationService.js:79-81`, `src/services/opportunitiesService.js:310`, `src/services/knowledgeBaseService.js:165`: all use template literals inside PostgREST `.or()` / `.ilike()`. activitiesService line 63 sanitizes search; most others do not. A name like `evil,assigned_to.is.null` could break OR semantics and widen results. Centralize sanitization in a helper.

### A.5 Sensitive Data Handling

#### 🟢 Low — `auditService` redacts password/token (good); but `firebase.js` ships a hardcoded API key + VAPID key
`src/lib/firebase.js:5-13`: Firebase API key + VAPID key are in source. Firebase web keys are designed to be public, but combined with no domain restriction at the Firebase console they'd allow abuse. Verify the Firebase project's authorized domains are restricted to your production hostnames.

#### 🟡 Medium — Meta Ads token stored in plain localStorage + system_config
`src/services/adsIntegrationService.js:16-26`: Facebook Graph API tokens live in `localStorage.platform_meta_ads_config` and `system_config.value` as plaintext. Any XSS or shared-browser scenario leaks them. Use Supabase Vault or move the sync server-side via an Edge Function with secret stored in env.

### A.6 Audit Gaps

#### 🟡 Medium — Several high-trust mutations don't log
- `notificationService.clearAll` (deletes all notifications) — no audit.
- `leadRecyclingService.runTemperatureDecay` — bulk mutation, no audit.
- `securityService.addIP / removeIP / toggleIPWhitelist` (`src/services/securityService.js:50-88`) — modifies IP allowlist, never audited.
- `securityService.savePasswordPolicy` (line 108-113) — changes auth policy, no audit.
- `financeService.createJournalEntry` audits create but not the journal lines.

#### 🟡 Medium — auditService falls back to wrong localStorage key
`src/services/auditService.js:146`: when supabase auth fails, reads `localStorage.platform_profile` — that key is never written anywhere; the real key is `platform_mock_user`. So fallback always yields a null user_id, meaning audit rows from any failed-auth path are unattributed.

### A.7 Public Exposure / Routing

#### 🟡 Medium — Several routes inside MainLayout lack a permission gate
`src/App.jsx:223-226`: `/dashboard`, `/contacts`, `/activities`, `/tasks` rely only on the outer `P.DASHBOARD` check. Every role has DASHBOARD, so /contacts is reachable by `marketing` (who only has `CONTACTS_VIEW_OWN`) and by `hr/finance` (who shouldn't see contacts at all). Same: `/calendar`, `/chat`, `/email`, `/whatsapp`, `/reports`, `/approvals`, `/notifications`, `/help`, `/knowledge-base`, `/announcements`, `/profile`. Gate each with the appropriate `P.*`.

#### 🟡 Medium — `print*` / export routes write user-controlled HTML into a new window via `document.write`
`src/utils/exportUtils.js:84-90`, `src/services/printService.js:510`, `src/services/reportExportService.js:209`, `src/pages/hr/EmployeeAttendanceSummary.jsx:413`, `src/pages/hr/PayrollPage.jsx:696`: tables interpolated as `${row[c.key]}` directly into HTML with no escaping. Any contact whose name contains `<script>` becomes XSS on print/export. Add HTML escaping helper.

### A.8 CSP

#### 🟢 Low — CSP allows `'unsafe-inline'` and `'unsafe-eval'` in scripts
`vercel.json:25`: `script-src 'self' 'unsafe-inline' 'unsafe-eval'`. `unsafe-eval` is required by some Sentry/Vite paths but `unsafe-inline` defeats the main protection CSP provides. Migrate to nonce-based CSP if feasible.

## B. Performance Audit (Full Report)

Built bundle: 10 MB total, biggest initial chunk `index-*.js` = 519 KB raw / 143 KB gz.

### B.1 N+1 Query Patterns

#### B.1.1 — `distributeLeadToAgents` — sequential INSERTs in a for loop
- `src/services/contactsService.js:905-963`
- Loops `targetUserIds` and does one `.insert(...).select().single()` PLUS a `logAudit` call **per agent** sequentially. Distributing 1 lead to 20 agents = 20 round-trips + 20 audit writes. With slow links this is several seconds.
- **Fix**: Build the clone array in JS, do one bulk `supabase.from('contacts').insert(arr).select(...)`, batch the audit log.

#### B.1.2 — `fetchMasterLeadsByOwner` — serial paginate-then-chunk loop
- `src/services/masterLeadsService.js:34-68`
- For a 7,000-contact agent: step 1 = 7+ sequential `range()` paginations to collect phones, step 2 = ~70 sequential `.in('phone', chunk)` round-trips. Page-load latency scales linearly with agent volume — easily 5-15s for a busy agent.

#### B.1.3 — `getTeamMemberIds` grandchildren loop
- `src/utils/teamHelper.js:18-23`
- For a sales_manager, loops each child team to fetch its grandchildren one at a time. With 10 child teams = 10 sequential queries. Called on nearly every page/load.

#### B.1.4 — `enrichOpps` chunkedIn pattern OK, but `recordAssignment` per-row in bulk reassign
- `src/pages/ContactsPage.jsx:636-644` calls `recordAssignment(c.id, ...)` for **every** selected contact (forEach without `await`/`Promise.all`). Each `recordAssignment` does an INSERT into activities + UPDATE on contacts. Bulk-reassigning 500 leads = 1,000 fire-and-forget requests racing each other; some will fail silently.

#### B.2.1 — `fetchOpportunityStats` selects all opps with no limit

#### B.2.2 — `loadSourceBreakdown` selects all contacts + opps with no limit

#### B.2.3 — `loadLeadsThisMonthByUser` un-paginated
